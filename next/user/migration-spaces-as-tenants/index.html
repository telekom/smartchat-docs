Migrating to Spaces-as-Tenants

The Spaces-as-Tenants update changes how Spaces work in SmartChat. Spaces are now strict multi-tenancy boundaries: each Space has its own isolated workspace items, files, and groups. This page describes what changes for end users and admins.

Migration tooling is still being built

There is no automatic migration in place yet. Existing data, settings, and content stay where they are — nothing is moved or deleted. Migration steps and tooling will be documented here once available.

What changed

Section titled “What changed”
BeforeAfter
Workspace items (collections, custom models, prompt templates)Only in My SpaceIn every Space; each Space has its own isolated set
Item portability between SpacesN/A (only My Space had them)Items don’t cross Space boundaries
RAG / Confluence-synced dataAuto-loaded as context on Space switchMust be manually attached via # and the {Space Name} - General collection
File upload (drag & drop)Only in My SpaceAvailable in all Spaces
GroupsOne group per Space synced into My Space (so you saw membership there)Synced into the respective Space; My Space sync stopped
Group identitySame group name → same group everywhereSame group name in different Spaces = different groups, with different permissions

For end users

Section titled “For end users”
  • Your chats are untouched. Chat history is preserved as-is; the chat list shows chats from all Spaces mixed together. Opening a chat automatically switches the active Space to the one the chat belongs to.
  • You can now upload files in any Space, not just My Space.
  • You can now create custom models, collections, and prompt templates in any Space. They stay in the Space where you create them and won’t follow you across Spaces.
  • If you used a custom model in My Space and want it in another Space, you’ll need to recreate it there — or use the migration tooling once we release it. Same for collections and prompt templates.
  • In Spaces with synced data (e.g. Confluence), RAG content no longer attaches automatically. In each new chat, add the relevant collections (e.g. {Space Name} - General) via # or the + icon to make the synced data available as context.
  • Sharing is per Space. A workspace item shared in one Space isn’t visible in another, even if both Spaces have a group with the same name. Items don’t move between Spaces; sharing applies only within the Space they live in.

Pick the right Space first

Before creating a workspace item, think about what other data you want to combine it with. Items can only reference other items in the same Space — a custom model in Space A can’t use a collection from Space B. Items don’t move between Spaces, so picking the right one up front saves rework.

If neither applies, just use My Space. Items are private by default regardless of Space; the choice mainly affects what you can combine and your maximum sharing scope.

For Space Owners and admins

Section titled “For Space Owners and admins”

Compliance: bounding the sharing scope

The choice of Space also bounds the maximum possible sharing scope of an item: an item in My Space can in principle be shared with every user of the system, while an item in a regular Space can only be shared with that Space’s members. This is a compliance feature — admins and Space Owners should give end users guidance on which Space to use for which kind of content. End users should not need to reason about this on their own.

Groups and permissions

Section titled “Groups and permissions”
  • Groups are now scoped per Space. A group with the same name in two Spaces is treated as two completely separate groups, each with its own membership and permissions. Sharing config you set in Space A does not apply in Space B, even if both have a group called engineering.
  • A user can be in groups of the same name in multiple Spaces and have different permissions in each — the permissions you see depend on the active Space.
  • The previous My Space group sync is stopped. New groups are created in their respective Space instead.

Open point — needs verification

Existing groups already synced into My Space by the previous behavior remain in place. Whether those legacy My Space groups continue to be updated by the upstream sync is an open question — confirm before relying on that membership for visibility.

Sharing

Section titled “Sharing”

Review your sharing config (which groups can see which workspace items) in each Space you own — sharing rules are now scoped per Space and need to be set up there.

Data sync

Section titled “Data sync”

Space-level data sync (S3 AutoSync, Confluence AutoSync, etc.) continues to operate at the Space level. The change is on the delivery side: synced content is no longer auto-attached to chats — users have to add the {Space Name} - General collection (which is auto-populated from the synced content) via # or the + icon.

What didn’t change

Section titled “What didn’t change”
  • Login, SSO, and user accounts
  • Chat history (preserved across the migration; mixed view across all Spaces in the sidebar)
  • The list of available LLMs in each Space
  • Per-Space Confidentiality and other Space-level configuration
  • Roles (User, Space Owner, Application Owner, Data Scientist, OpenWebUI-Admin)
